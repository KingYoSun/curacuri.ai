# 日本語Discord分類ラベル v0

## 目的

AIプロダクト公式Discordの投稿を、運営者が短時間で確認できる形に整理する。この taxonomy は Phase
0 の検証用であり、アプリAPI、DB schema、Redis queue 名は定義しない。

## ラベル運用方針

- 1投稿に複数ラベルを付けてよい。
- ラベルはユーザーの評価ではなく、投稿内容の運営確認カテゴリである。
- 確信度が低い場合は `雑談` と重要ラベルの併記を避け、理由に曖昧さを書く。
- `炎上兆候`、`誤情報可能性`、`ルール違反候補` は断定ではなく運営確認候補として扱う。
- Phase
  1 では通知疲れを避けるため、管理者通知は高重要度の質問・バグ報告・不満増加・誤情報可能性・炎上兆候に絞る。

## 重要度

| 値         | 意味                                                     |
| ---------- | -------------------------------------------------------- |
| `low`      | 週報に載せなくてもよい、または集計だけでよい             |
| `medium`   | 週報候補。急ぎではないが傾向として拾う                   |
| `high`     | 管理者が確認すべき投稿または話題                         |
| `critical` | 早期確認が必要。障害、強い不満、誤情報拡散、炎上兆候など |

## MVP v0.1 基本ラベル

### 質問

- 定義: 使い方、仕様、設定、料金、制限、エラーの意味などを尋ねている投稿。
- 含める例: 「Webhookの設定ってどこからできますか？」
- 含めない例: 「Webhookが壊れている気がする」は、再現情報があれば `バグ報告` を優先する。
- 曖昧な場合: 末尾が疑問形でなくても、回答を求めている内容なら付ける。

### 未回答質問

- 定義: 質問のうち、一定時間回答がなく、運営またはコミュニティ回答が必要そうな投稿。
- 含める例: 「昨日から返信がないのですが、この制限は仕様ですか？」
- 含めない例: 直後に自己解決している投稿。
- 曖昧な場合: 単体投稿だけでは判断しきれないため、文脈がない場合は `質問` のみにする。

### 公式回答待ち

- 定義: 一般ユーザーでは確定回答しづらく、仕様・方針・障害状況など公式確認が必要な投稿。
- 含める例: 「この料金変更は既存ユーザーにも適用されますか？」
- 含めない例: ドキュメントに明確な回答がある使い方質問。
- 曖昧な場合: 法務、料金、障害、ロードマップ、仕様変更に関わるなら付ける。

### バグ報告

- 定義: 期待と異なる挙動、エラー、クラッシュ、再現可能な不具合を報告している投稿。
- 含める例: 「保存ボタンを押すと500になり、再読み込みしても復旧しません」
- 含めない例: 使い方が分からないだけの投稿。
- 曖昧な場合: エラーコード、環境、再現手順、複数人の同様報告があれば優先する。

### 要望

- 定義: 機能追加、改善、仕様変更、連携追加などを求める投稿。
- 含める例: 「Slackにも同じ通知を飛ばせるようにしてほしい」
- 含めない例: 不具合修正を求める投稿は `バグ報告` を優先する。
- 曖昧な場合: 「できたら嬉しい」「欲しい」「対応予定ありますか」は要望候補にする。

### 不満

- 定義: 体験、価格、仕様変更、対応速度、品質へのネガティブな感情や失望が含まれる投稿。
- 含める例: 「アップデート後から前より使いづらくなった」
- 含めない例: 冷静な改善提案のみの投稿。
- 曖昧な場合: 皮肉、遠回しな表現、絵文字だけの落胆も文脈に応じて拾う。

### 称賛

- 定義: プロダクト、アップデート、サポート、コミュニティへの肯定的な反応。
- 含める例: 「新しい検索、かなり速くなって助かります」
- 含めない例: 問題報告の前置きとしての軽い肯定のみ。
- 曖昧な場合: 週報のポジティブ反応として拾えるなら付ける。

### 雑談

- 定義: 運営判断、FAQ、Issue化、通知に直結しない会話。
- 含める例: 「今日のアップデート名かっこいい」
- 含めない例: 雑談の中に具体的な不具合や要望が含まれる投稿。
- 曖昧な場合: 他ラベルに該当しなければ雑談にする。

## 運営確認候補ラベル

### 炎上兆候

- 定義: 不満や不信が短時間に増え、拡散・対立・運営不信につながる可能性がある投稿。
- 含める例: 「また説明なしで変えたの？前も同じことしてたよね」
- 含めない例: 単発の軽い不満。
- 曖昧な場合: 初期は断定通知せず、`不満` と併記して参考シグナルにする。

### 誤情報可能性

- 定義: 仕様、料金、障害、セキュリティなどについて、誤解または未確認情報が広がりそうな投稿。
- 含める例: 「無料プランは来月で全部終了らしいです」
- 含めない例: 明確に質問として確認している投稿。
- 曖昧な場合: 公式回答が必要なら `公式回答待ち` も付ける。

### ルール違反候補

- 定義: サーバールールや安全方針に触れる可能性がある投稿。
- 含める例: APIキーや個人情報を投稿している、禁止された宣伝をしている。
- 含めない例: 単なる強い意見。
- 曖昧な場合: Phase 1 では自動処分せず、管理者確認候補に留める。

### 高価値UGC

- 定義: 他ユーザーの役に立つ事例、テンプレート、検証結果、解説、作品、ワークフロー。
- 含める例: 「Webhook設定で詰まったので手順をまとめました」
- 含めない例: 完成物の一言共有だけで再利用性が低いもの。
- 曖昧な場合: FAQ、Docs、事例記事に転用できそうなら付ける。

### 新規参加者の困りごと

- 定義: 初回設定、用語、導線、権限、ドキュメント不足でつまずいている投稿。
- 含める例: 「どのチャンネルで質問すればいいか分かりません」
- 含めない例: 高度なAPI仕様質問。
- 曖昧な場合: オンボーディング改善に使えるなら付ける。

### 古参の重要指摘

- 定義: 長期利用者や詳しいユーザーが、仕様変更、文化、品質、過去経緯に基づき重要な指摘をしている投稿。
- 含める例: 「前回もこの変更で混乱したので、今回は移行期間を置いた方がいいです」
- 含めない例: 単なる昔話。
- 曖昧な場合: 運営判断や移行計画に影響しそうなら付ける。
