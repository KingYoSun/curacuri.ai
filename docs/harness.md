# ハーネス

curacuri.ai は Codex 中心の開発を前提に、最小実行可能ハーネスを採用する。ここでの「最小」は開発基盤の範囲を小さく保つという意味であり、プロダクトを薄い最小実装で公開する方針ではない。アプリケーションコードが増える前に、リポジトリを読みやすく、テスト可能で、決定的なコマンドで運用しやすい状態にする。

## ローカルワークフロー

1. 現在の入口として `AGENTS.md` を読む。
2. `pnpm install` で依存関係をインストールする。
3. 変更範囲を絞って作業する。
4. 引き渡し前に `pnpm check` を実行する。

## 品質ゲート

- `pnpm typecheck` は TypeScript を検証する。
- `pnpm lint` は ESLint でコードを検証する。
- `pnpm format:check` は Prettier でフォーマットを検証する。
- `pnpm test` は Vitest を実行する。
- Lefthook は pre-commit で `pnpm check` を実行する。

## エージェントワークフロー

- `proposal.md` をプロダクト文脈として扱う。
- `docs/adr/` を長く残る技術判断として扱う。
- スタック、アーキテクチャ、データ境界、運用方針を変える判断では ADR を追加または更新する。
- 文章は短く保ち、強制すべきルールはテスト・型・リンター・フックへ移す。

## 現在の境界

このハーネスでは、アプリケーションルート、データベーススキーマ、Redis queue 名、Discord
Bot の振る舞い、デプロイ構成を定義しない。これらの判断は、関連する実装作業と一緒に導入する。
