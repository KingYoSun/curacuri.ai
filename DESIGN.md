# DESIGN.md

このリポジトリのダッシュボード UI は、Ben
Shneiderman 氏の「インターフェイスデザインの8つの黄金律」を基本原則として扱う。ここでは一般論ではなく、curacuri.ai の運営者向け管理画面で守るべき実装判断をまとめる。

## 1. 一貫性を保つ

- 同じ意味の操作には同じラベル、配置、見た目を使う。
- API や DB の内部値をそのまま見せず、運営者が理解できる日本語ラベルに変換する。
- 保存、承認、却下、再実行、フィードバックなどの操作結果表示を画面全体で揃える。
- フォーム、一覧、詳細、空状態、エラー状態の構造を画面間で揃える。

## 2. 幅広い利用者に対応する

- 初回運用者には状態と次の行動が分かり、慣れた運用者には素早く処理できる導線を用意する。
- キーボード操作、フォーカス表示、十分なコントラスト、読み上げ可能なラベルを保つ。
- 大量データでは検索、絞り込み、並び替え、ページングまたは「続きを見る」を用意する。
- モバイル幅でも主要操作と状態が崩れないようにする。

## 3. 有益なフィードバックを返す

- 保存、承認、却下、再実行、インポート、レポート生成には処理中、成功、失敗の状態を表示する。
- 失敗時は「何が」「なぜ」「次に何をすべきか」が分かる文言にする。
- 長い処理や一括処理では、対象範囲と処理件数を表示する。
- エラーは画面上部だけでなく、可能なら該当する操作や行にも紐づける。

## 4. 操作完了を明確にする

- 操作後は完了メッセージ、更新時刻、変更後の状態などで区切りを示す。
- 一覧行の状態変更は、変更後の status がすぐ確認できるようにする。
- レポート生成や LLM 再実行のような非同期処理は、受付完了と処理完了を区別して表示する。
- 保存前の draft と保存済み状態を混同させない。

## 5. エラーを予防する

- 副作用が大きい操作には確認を入れる。例: LLM 一括再実行、自動返信の承認、却下。
- 数値入力には最小値、最大値、ステップ、単位を明示する。
- チャンネル ID、ラベル、カテゴリなどは自由入力だけにせず、可能なら候補選択や検証を用意する。
- 不正な入力や矛盾する設定は送信前に検出する。

## 6. 取り消しを許す

- 承認、却下、dismiss、feedback などの状態変更は、可能な限り取り消しまたは再変更できるようにする。
- 取り消しできない操作は、実行前に影響範囲を明示する。
- 設定フォームでは未保存変更の破棄、保存前確認、変更差分の表示を検討する。
- refresh や他の操作で編集中の draft を黙って上書きしない。

## 7. ユーザー主導感を保つ

- 自動更新や一括処理は、運営者の編集中データや画面位置を壊さない。
- 画面が勝手に状態を変える場合は、理由とタイミングを分かるようにする。
- 主要操作は明示的なボタンで実行し、暗黙の保存や暗黙の送信を避ける。
- フィルタ、期間、対象チャンネルなど、運営者が分析範囲を制御できる UI を用意する。

## 8. 短期記憶負荷を減らす

- ID、内部 status、confidence などは、意味が分かる補助情報と一緒に表示する。
- 投稿、分類、通知、FAQ、自動返信、週次レポートは、関連する文脈へ移動できるようにする。
- よく使う判断材料は同じ画面内に置き、別ページや別ファイルを覚えさせない。
- 空状態には「何もない」だけでなく、必要なら次の行動を示す。

## ダッシュボード変更時のチェック

- 操作後に処理中、成功、失敗が見えるか。
- 誤操作しやすいボタンに確認、無効化、取り消しのいずれかがあるか。
- 表示文言が運営者向けの日本語になっているか。
- 内部値を出す場合、その意味を画面内で理解できるか。
- フォーム入力が不正値を事前に防げるか。
- refresh、再取得、一括処理で未保存の編集内容が失われないか。
- 一覧が増えたときに検索、絞り込み、ページングの導線があるか。
- モバイル幅とキーボード操作で主要操作が使えるか。
