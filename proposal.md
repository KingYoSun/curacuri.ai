# curacuri.ai（クラクリAI）企画書 v2

## 1. 概要

**curacuri.ai（クラクリAI）** は、少数精鋭で公式Discordを長期運営するチームのための、**AI書記・受付・運営補佐キャラクター**である。

表向きには、Discordサーバーに常駐するAIキャラクターとして、質問・要望・不満・バグ報告・盛り上がりを自然に受け止める。

裏側では、公式Discordの投稿を整理し、週次レポート・重要通知・FAQ候補・VOC分析として運営者に届けることで、運営者の「全部読まなければならない」という認知負荷を下げる。

curacuri.ai は、単なるDiscord Botでも、AI FAQ Botでも、モデレーションBotでもない。

> 公式Discordを少人数で長期運営するための、AI書記キャラクターである。  
> 1週間分の声を整理し、重要な質問・不満・要望・バグ報告を見落とさない状態を作ることで、運営者の認知負荷を下げ、創作・開発・コミュニティ育成に集中できる環境を提供する。  
> 長期的には、コミュニティの判断・FAQ・文化・歴史を蓄積する「共同体の第二の脳」へ進化する。

## 2. コンセプト

### 2.1 一言で

> 公式Discordの声を整理し、小規模運営者の認知負荷を下げるAI書記キャラクター。

### 2.2 提供する状態

curacuri.ai が提供するのは、「AIが喋るDiscord Bot」ではなく、以下の状態である。

- 全部読まなくても、重要な声を見落とさない
- 質問・要望・不満・バグ報告が自然に整理される
- 週に一度、Discordの状況を短時間で把握できる
- 運営者が雑務ではなく、創作・開発・意思決定に集中できる
- 低コストでも公式コミュニティの運営品質を維持できる
- ユーザーの声が、流れて消えず、運営判断に接続される
- 長期的に、コミュニティの記憶が蓄積されていく

### 2.3 キャッチコピー案

#### 汎用

- 公式Discordに、AI書記を。
- 1週間分のDiscordを、5分で把握する。
- ユーザーの声を、流さず、拾い、運営に届ける。
- Discordを、ただの雑談場から「運営資産」に変える。
- 認知負荷を下げて、創作と開発に集中できる運営環境を。

#### 小規模運営者向け

- 1人で回す公式Discordに、AI書記を。
- 少人数運営でも、拾うべき声を見落とさない。
- 全部読まなくても、今週のDiscordが分かる。
- コミュニティ運営の雑務を減らし、つくる時間を取り戻す。

#### AIプロダクト向け

- 技術質問・不具合報告・要望を、公式Discordから自動で整理する。
- Discordの検証ログを、Docs・FAQ・Issueにつなげる。
- ユーザーの声を、プロダクト改善に変えるAI書記。

#### 運営代行会社向け

- 複数Discordの重要投稿抽出とレポート作成をAIで効率化する。
- クライアント向けDiscord運営レポートを、AI書記が下書きする。
- 人力監視の品質を安定させるCommunity Ops補助ツール。

### 2.4 第二の脳という長期ビジョン

curacuri.ai は、初期段階では「公式DiscordのAI書記」として提供する。

ただし、長期的には以下を蓄積し、コミュニティ運営の「第二の脳」へ進化する。

- 過去の質問
- 重要な要望
- 繰り返された不満
- バグ報告の履歴
- FAQの変遷
- 運営判断の履歴
- コミュニティの文化
- 新規参加者がつまずくポイント
- 古参が大事にしている文脈
- サーバーの歴史

初期訴求では具体的なAI書記として分かりやすく伝え、長期ビジョンとして「共同体の第二の脳」を提示する。

## 3. 背景

ゲーム、AIプロダクト、VTuber、クリエイター、VRChat、UGCサービスなどの領域では、公式Discordサーバーがユーザー接点として重要になっている。

Discordには以下の強みがある。

- コアユーザーが集まりやすい
- ユーザーとの距離が近い
- 不満・要望・バグ報告が早く出る
- ファン同士の関係性が生まれる
- 公式とユーザーの共同体感を作りやすい
- ベータテストや検証コミュニティと相性が良い
- 小規模チームでも始めやすい

一方で、公式Discord運営には以下の問題がある。

- 質問が流れて見落とされる
- バグ報告が雑談に埋もれる
- 要望や不満が整理されない
- 公式回答すべき話題を検知できない
- 炎上や運営不信の兆候を見落とす
- 新規参加者が入りづらい
- 古参だけの空気になりやすい
- 運営担当者が全チャンネルを読み切れない
- Discord運営が属人化する
- Discordの情報がプロダクト改善やCSに接続されない
- 専任コミュニティマネージャーや運営代行を置く予算がない

海外には、Mava、CommunityOne、GGWP、Thena、Pylon など、Discord向けAIサポート・分析・モデレーション・チケット管理の競合が存在する。

しかし、日本のゲーム/VTuber/AI/クリエイター系公式Discordでは、こうしたツールが十分に認知・導入されていないケースが多い。

このギャップには、以下の理由がある。

- Discord運営が日本企業内で業務カテゴリとして未成熟
- 導入担当部署・予算・責任範囲が曖昧
- 海外ツールが日本語コミュニティの空気に最適化されていない
- 高機能AI Botは公式事故リスクが怖い
- 人力運営代行は小規模チームには重い
- 低予算のクリエイター/小規模ゲーム/AIプロダクトには高額SaaSが合わない
- DockerやAPIキー設定を前提にしたOSSは非エンジニアには難しい
- 「サポートツール」ではなく「場の運営支援」が必要

### 3.1 なぜ今必要なのか

AIによって、少人数でもプロダクト開発・運営・コンテンツ制作が可能になりつつある。

一方で、公式Discordのようなコミュニティ運営は、ユーザー数や投稿量が増えるほど、人間の認知負荷を直接増やす。

少数精鋭チームにとって、以下を同時に担うことは長期的に大きな負担になる。

- 開発
- 創作
- CS
- 広報
- コミュニティ運営
- バグ対応
- 要望整理
- ファン対応
- 炎上予防
- ドキュメント整備

AI時代には、少人数で作れるものが増える一方で、ユーザーとの接点も増える。  
その結果、**作る力よりも、継続して運営する認知負荷がボトルネックになる**。

curacuri.ai は、この「少人数で長期運営する時代の認知負荷問題」を解決する。

## 4. 対象者が抱える根本的制約

小規模・個人・少数精鋭で公式Discordを長期運営する場合、以下の構造的制約が存在する。

### 4.1 認知負荷の上限

1人または少人数で、全チャンネル・全スレッド・全質問を継続的に把握することは難しい。

特に、以下のような状況では認知負荷が急増する。

- アップデート直後
- 不具合発生時
- 新機能リリース時
- 炎上気味の話題が発生した時
- 新規ユーザーが増えた時
- イベント開催後
- 外部SNSから流入が増えた時

### 4.2 見落としリスク

公式Discordでは、重要な声が雑談に埋もれやすい。

- バグ報告
- 要望
- 不満
- 仕様質問
- 誤解
- 称賛
- 離脱兆候
- 初心者のつまずき
- 古参からの重要な指摘

これらは、拾えればプロダクト改善や信頼形成につながるが、見落とすと「運営は見ていない」という不信感につながる。

### 4.3 モチベーションの摩耗

小規模運営者にとって、Discord運営は最初は楽しい。

しかし、運営が続くほど以下のような雑務が積み上がる。

- 同じ質問への対応
- 不満の確認
- バグ報告の整理
- 要望の優先順位付け
- 初心者対応
- 古参との温度差調整
- 告知
- ルール整備
- FAQ更新

これらが積み重なると、本来やりたかった創作・開発・企画に使える集中力が削られる。

curacuri.ai は、運営者の情熱を摩耗させる雑務を軽減し、長期的に続けられる運営環境を作る。

### 4.4 低コストな運営品質

専任コミュニティマネージャーや運営代行を置けない場合でも、公式Discordには一定の運営品質が求められる。

- 質問を放置しない
- 不具合報告を拾う
- 要望を整理する
- 不満を無視しない
- 公式回答が必要な話題を見つける
- コミュニティの温度感を把握する
- ユーザーの声をプロダクト改善へつなげる

小規模運営者に必要なのは、大企業向けの重いコミュニティ管理SaaSではなく、**低コストで導入できるAI書記**である。

### 4.5 curacuri.ai の役割

curacuri.ai は、これらの制約に対して、AI書記として「読む・整理する・届ける」を肩代わりし、小規模運営者の認知負荷を下げる。

```text
人間がやるべきこと:
- 判断する
- 返信する
- 方針を決める
- 創作する
- 開発する
- ユーザーと向き合う

curacuri.ai が補助すること:
- 読む
- 分類する
- 要約する
- 重要な声を拾う
- レポートする
- FAQ候補を出す
- 見落としを減らす
```

## 6. 課題仮説

### 6.1 公式Discord運営者の状態

公式Discord運営者は、以下の状態に陥りやすい。

```text
Discordは盛り上がっている。
しかし、何を拾うべきか分からない。
全部読む時間はない。
質問も要望も不満もバグ報告も流れていく。
運営が反応しないと、ユーザーは「見られていない」と感じる。
結果として、せっかくのコアユーザーの声が資産にならない。
```

### 6.2 現場で本当に欲しいもの

現場が本当に欲しいのは、「よく喋るAI Bot」ではない。

欲しいのは以下である。

- 見落としてはいけない投稿の抽出
- 公式回答が必要な質問の通知
- バグ報告・要望・不満の分類
- ユーザーの温度感の把握
- FAQ候補の生成
- 日次/週次レポート
- 運営判断ログ
- CS/開発/広報への接続
- ユーザーに不自然に見えない受付キャラクター
- 事故りにくいAI導入
- 導入時に「監視されている」と感じさせない説明設計

### 6.3 課題の本質

curacuri.ai が扱う課題は、単なる作業効率化ではない。

本質は以下である。

```text
公式Discordの運営負荷は、
投稿量に比例して増える。

しかし、小規模運営者の認知資源は増えない。

その結果、
重要な声の見落とし、
運営者の疲弊、
コミュニティへの反応遅れ、
ユーザー不信、
プロダクト改善機会の損失
が発生する。
```

curacuri.ai は、この認知資源の不足を補うAI書記である。

## 7. 価値提案

curacuri.ai の価値は、3層構造で整理できる。

| 層 | 価値 | 具体的な提供体験 |
|---|---|---|
| 表層 | 業務効率化 | 重要投稿の抽出、週次レポート、FAQ候補生成 |
| 中層 | 認知負荷低減 | 全部読まなくても安心できる状態を作る |
| 深層 | 長期運営の持続可能性 | 雑務による疲弊を減らし、創作・開発・コミュニティ育成を続けられる |

### 7.1 表層価値: 業務効率化

- 重要投稿を自動抽出する
- 質問・要望・不満・バグ報告を分類する
- FAQ候補を生成する
- 日次/週次レポートを作る
- 管理者向け通知を出す
- GitHub Issues / Notion / Slack などへ接続する

### 7.2 中層価値: 認知負荷低減

運営者は、Discordの全投稿を読み続ける必要がなくなる。

```text
Before:
全部読まないと不安。
何か見落としている気がする。
どの投稿に対応すべきか分からない。

After:
週次レポートを読めば全体像が分かる。
重要な投稿は通知される。
未回答質問や要望が整理されている。
```

これにより、運営者は「全部見なければならない」という心理的負担から解放される。

### 7.3 深層価値: 長期運営の持続可能性

curacuri.ai は、運営者のモチベーションを守るためのプロダクトでもある。

- 雑務で創作意欲を削られない
- 不満や問い合わせに飲み込まれない
- ユーザーの声を前向きに扱える
- 長期的に公式コミュニティを続けられる
- 運営判断やFAQが蓄積され、将来の負担が減る

単なる「サポート削減」ではなく、**小規模チームが長く続けるための運営基盤**である。

## 8. プロダクト方針

### 8.1 基本思想

curacuri.ai の根本思想は、**小規模運営者の認知負荷を最小化し、重要なユーザーの声を確実に拾いながら、長期的な情熱と運営品質を両立させること**である。

自動応答を主役にせず、まずは「運営者の認知を補助するAI書記」として機能する。

キャラクター性は、単なる演出ではなく、負担を軽減する相棒としての信頼感と親しみやすさを生むために設計する。

### 8.2 初期は読み取り専用から始める

curacuri.ai は、最初からユーザーに積極的に話しかけない。

初期は、読み取り専用AI書記として以下を行う。

- 指定チャンネルを読む
- 重要投稿を分類する
- 週次レポートを作る
- 管理者に通知する
- FAQ候補を出す
- ユーザーには発言しない

これにより、AI事故や監視感を抑えながら、運営者に価値を届ける。

### 8.3 段階設計

```text
Level 1: Read-only AI書記
- 指定チャンネルを読み取る
- 重要投稿を分類する
- 管理者専用チャンネルに通知する
- 日次/週次レポートを作る
- ユーザーには発言しない

Level 2: AI受付キャラクター
- 「記録しました」
- 「運営確認待ちです」
- 「似た質問はこちらです」
- 「このチャンネルが適切です」
- 公式見解は勝手に言わない

Level 3: 承認付き公式案内
- 回答案を生成する
- 管理者が承認して投稿する
- FAQ/RAGベースで出典付き回答を行う
- 公式回答ログを保存する

Level 4: Community Ops Dashboard
- VOC分析
- 炎上兆候検知
- 参加者の温度感分析
- 新規参加者の孤立検知
- 古参化/閉鎖性の兆候検知
- プロダクト改善候補の抽出
- 外部ツール連携

Level 5: 共同体記憶OS
- コミュニティの歴史
- 過去の運営判断
- よく出る論点
- 重要イベント
- FAQ変遷
- サーバー文化
- 参加者の貢献履歴
```

## 9. MVP方針

### 9.1 MVPの最重要方針

MVPでは、OSS/self-hostだけでなく、最初から以下を同時に用意する。

```text
1. OSS/self-host版
2. Hosted Free版
3. 導入デモ用サンプルDiscord
```

OSSだけでは、非エンジニアのクリエイター・VTuber・小規模運営者にとって導入が難しい。

そのため、最初から **1クリックで試せるHosted版** を用意する。

### 9.2 MVPで提供する中核体験

MVPで提供する中核体験は以下である。

> 全部読まなくても、今週のDiscordで拾うべき声が分かる。

具体的には、

> 1週間分のDiscordを、5分で把握できる。

という体験を作る。

### 9.3 MVP v0.1 優先順位

1. **週次運営レポート**
   - 1週間分のDiscordを5分で把握できる体験を作る
2. **重要投稿の自動分類**
   - 質問・要望・不満・バグ報告・称賛を分類する
3. **管理者向け重要通知**
   - 即時対応が必要なものだけ通知する
4. **未回答質問の検知**
   - 放置されている質問を見つける
5. **FAQ候補生成**
   - 繰り返し出る質問を整理する

## 10. 分類精度検証計画

### 10.1 なぜ分類精度検証が重要か

curacuri.ai の初期価値は、投稿分類と重要投稿抽出に依存する。

以下の精度が低い場合、ユーザーはすぐに離脱する。

- 質問の検出
- 未回答質問の検出
- バグ報告の検出
- 要望の検出
- 不満の検出
- 炎上兆候の検出
- 雑談との切り分け

特に日本語Discordでは、以下が難しい。

- スラング
- 絵文字
- 略語
- 内輪ネタ
- 皮肉
- 遠回しな不満
- 文脈依存の発言
- VTuber/ゲーム固有の言い回し
- スクリーンショットや画像添付に依存する報告

### 10.2 検証データ

初期検証では、以下のログを用いる。

- 自前のテストDiscordログ
- 協力者のAIプロダクトDiscordログ
- インディーゲーム/創作コミュニティの匿名化ログ
- 公開可能なサンプル投稿セット
- 手動作成した日本語Discord風データセット

### 10.3 評価指標

最低限、以下を測定する。

- Precision
- Recall
- F1
- 管理者が「有用」と評価した通知率
- 誤通知率
- 見逃し率
- 週報の有用性評価
- FAQ候補の採用率

### 10.4 初期合格ライン

MVP公開前の目安。

```text
質問検出: F1 0.80以上
バグ報告検出: F1 0.75以上
要望検出: F1 0.75以上
不満検出: F1 0.70以上
炎上兆候検出: 初期は参考値扱い
管理者通知の有用率: 70%以上
週次レポートの有用率: 70%以上
```

炎上兆候検知は誤検出リスクが高いため、初期は「参考シグナル」として扱い、断定的な通知はしない。

### 10.5 精度改善サイクル

- 管理者が通知を「有用/不要/誤分類」と評価
- 評価結果をプロンプト・ルール・分類器に反映
- サーバーごとの語彙・チャンネル文脈を学習
- 高リスク分類は人間確認前提にする
- 精度レポートを管理画面に表示する

## 11. 監視感ゼロ設計

### 11.1 問題意識

curacuri.ai はDiscord投稿を読み取り、分類・要約する。

そのため、ユーザーから見ると「AIに監視されている」と感じられるリスクがある。

特にファンコミュニティでは、自由な雑談の場であることが重要であり、導入説明を誤ると不信感や反発につながる。

### 11.2 基本方針

curacuri.ai は「監視」ではなく、以下として説明する。

> ユーザーの声を運営が見落とさないためのAI書記。

重要なのは、ユーザーを評価・監視・採点するのではなく、以下を目的にすることである。

- 質問を放置しない
- 要望を流さない
- バグ報告を見落とさない
- 不満を無視しない
- 運営が声を拾いやすくする

### 11.3 初期設定で必ず提供するもの

- ユーザー向け導入告知テンプレート
- 対象チャンネル一覧の表示
- 分析対象外チャンネル設定
- DMは読まないことの明記
- 保存期間設定
- 取得・保存する情報の説明
- 削除リクエスト対応方針
- Botの発言範囲の明示
- 自動回答しないモードの明示

### 11.4 ユーザー向け告知テンプレート例

```text
このサーバーでは、運営が質問・要望・不具合報告を見落とさないように、
AI書記「クラクリAI」を導入しています。

クラクリAIは、指定された公開チャンネルの投稿から、
運営が確認すべき質問・要望・不具合報告などを整理します。

クラクリAIは以下を行いません。

- DMの読み取り
- ユーザーの評価・採点
- 自動BAN
- 未確認の公式回答
- 個人を対象にした監視

目的は、みなさんの声を運営がより確実に受け取れるようにすることです。
```

### 11.5 UI/文言の原則

避ける表現。

- 監視
- スコアリング
- 危険ユーザー
- 問題ユーザー

使う表現。

## 13. キャラクター設計

### 13.1 キャラクターの役割

curacuri.ai のキャラクター性は、単なる会話演出ではない。

役割は以下である。

- コアユーザーにとっては記録係
- 開発者にとってはVOCフィルター
- コミュニティにとっては場の記憶

### 13.2 キャラクターの位置づけ

キャラクターは、運営者の認知負荷を減らす相棒である。

```text
冷たい監視Botではない。
無責任に喋るAIキャラでもない。
場の声を丁寧に拾い、運営に届ける書記である。
```

- ギルド受付
- 研究助手
- 管制AI
- 書記官
- マネージャー

- 公式案内係
- 町の記録係
- 編集部員
- 学園の事務員
- AI実験室ナビゲーター

### 13.4 安全制約

キャラクターは以下を行わない。

- 未確認の公式見解を言わない

- モデレーション処分を勝手に実行しない

- ユーザーをスコアリングしない
- DMを読まない
- 公式回答は承認制にする
- 情報源がない回答は「確認中」とする

### 14.3 コスト対策

```text

2. ルール/軽量分類で一次選別
3. 重要候補のみLLM分類
4. スレッド/時間窓単位で要約
5. レポート生成はバッチ実行
6. 保存期間と分析対象を制限可能にする
```

### 18.3 追加差別化

#### 認知負荷特化設計

小規模運営者が「全部見なければならない」状態から解放されることを最優先する。

#### モチベーション維持

雑務や問い合わせ整理による疲弊を減らし、創作・開発・コミュニティ育成に集中できる状態を作る。

#### 長期運営支援

一時的な問い合わせ削減ではなく、数ヶ月〜数年単位で場の記憶を蓄積する。

#### 共同体記憶への進化

FAQや週報を積み重ね、サーバーの歴史・判断・文化を記録していく。

### 18.4 競合に対する防衛線

海外競合やDiscord公式がAI機能を強化した場合に備え、以下を防衛線とする。

- 日本語コミュニティ運営テンプレート
- 日本語Discord分類データセット
- 国内導入事例
- キャラクター/世界観テンプレート
- 監視感ゼロ導入ノウハウ
- 運営代行会社向けワークフロー
- 公式Discord運営教育コンテンツ
- OSSコミュニティ
- 共同体記憶の思想と実装

## 19. リスクと対策

### 19.1 競合が日本語対応を強化する

#### リスク

Mava、CommunityOne、Discord公式などが日本語対応・AI運営支援機能を強化する。

#### 対策

- 日本語Discord特化の分類精度を公開
- 国内導入事例を早期に作る
- 監視感ゼロ設計を差別化する
- キャラクター/世界観テンプレートを充実させる
- 運営代行会社との関係を作る

### 19.2 分類精度が低い

#### リスク

誤分類が多いと「使えない」と判断される。

#### 対策

- MVP前に分類精度検証を行う
- 管理者フィードバックUIを作る
- サーバー別カスタム分類を可能にする
- 確信度の低い通知は断定しない
- 重要投稿のみ通知し、通知疲れを防ぐ

### 19.3 週次レポートが薄い

#### リスク

MVPの中核体験である週次レポートが表面的だと、認知負荷低減の価値を感じられない。

#### 対策

- 週報のテンプレートをターゲット別に用意する
- 管理者が有用/不要を評価できるようにする
- 「主要トピック」「未回答質問」「不満」「要望」「バグ報告」を必ず含める
- 具体的な投稿リンクを添える
- 次に取るべきアクションを提示する

### 19.4 LLM APIコストが高い

#### リスク

常時分類により運用コストが増える。

#### 対策

- ルールベース一次フィルタ
- 小型モデル分類
- バッチ処理
- 重要候補のみLLM処理
- Hosted Freeの処理数制限
- APIキー持ち込み
- OpenAI互換/ローカルLLM対応

### 19.5 セットアップが難しい

#### リスク

OSSだけでは非エンジニアが導入できない。

#### 対策

- Hosted FreeをMVPから提供
- Discord OAuthで簡単導入
- サンプル設定を提供
- 導入ウィザードを作る
- テンプレートから始められるようにする

### 19.6 監視感・不信感が出る

#### リスク

ユーザーが「AIに監視されている」と感じる。

#### 対策

- 導入告知テンプレート
- 対象チャンネルの明示
- DM非対応
- 保存期間設定
- 分析対象外チャンネル設定
- ユーザースコアリングをしない
- 「監視」ではなく「声を届ける」文脈で説明する

### 19.7 収益化が難しい

#### リスク

低予算クリエイター層は月額課金しにくい。

#### 対策

- 初期はHosted Free/OSSで普及
- 収益化は運営代行会社、AIプロダクト、ゲーム会社を優先
- Agencyプランを早期に設計
- 導入支援・カスタムレポート・Hosted運用で課金する

### 19.8 Discord ToS・プライバシー対応

#### リスク

ユーザー投稿の保存・分析が規約やプライバシー上の懸念になる。

#### 対策

- Discord Developer Policyを確認
- プライバシーポリシー整備
- 保存期間設定
- 削除対応
- 対象チャンネル明示
- DM非対応
- ユーザー向け告知テンプレート
- 管理者向け導入チェックリスト

### 19.9 ポジショニングが抽象化しすぎる

#### リスク

「第二の脳」「認知負荷低減」を前面に出しすぎると、何をするプロダクトなのか分かりにくくなる。

#### 対策

初期の説明では、以下の具体的価値を前面に出す。

- 公式DiscordのAI書記
- 1週間分のDiscordを5分で把握する
- 重要な質問・不満・要望・バグ報告を見落とさない

「第二の脳」は長期ビジョンとして扱う。

## 23. 独自性

curacuri.ai の独自性は、単体機能ではなく、以下の統合にある。

```text
読み取り専用AI書記
+ 日本語Discord分類
+ 認知負荷低減
+ 週次運営レポート
+ 監視感ゼロ設計
+ VOC分析
+ FAQ候補
+ 公式運営の安全設計
+ キャラクターUI
+ OSS/self-host
+ Hosted Free
+ Agency向けB2B
+ 共同体記憶
```

特に重要なのは、以下の思想である。

> Botではなく、場に宿るAI書記。  
> FAQではなく、共同体の記憶。  
> 自動応答ではなく、運営の見落としを減らす。  
> 監視ではなく、ユーザーの声を運営に届ける。  
> サポート削減ではなく、公式コミュニティの運営品質を上げる。  
> 便利ツールではなく、小規模運営者の認知負荷を下げる相棒。

## 24. 将来的なビジョン

curacuri.ai は、最終的には以下を目指す。

```text
すべての公式Discordサーバーに、
その場の歴史・空気・ルール・ユーザーの声を覚えている
AI書記キャラクターがいる状態。
```

このAIキャラクターは、単なるチャット相手ではない。

- 新規参加者には案内役
- 常連には記録係
- 運営者には副操縦士
- 開発者にはVOCフィルター
- CSにはFAQ生成補助
- 広報には炎上兆候検知
- コミュニティには記憶の器

として機能する。

長期的には、Discordだけでなく、Slack、Forum、GitHub Discussions、SNS、Webコミュニティなどにも拡張し、各コミュニティに「場の人格」と「共同体記憶」を与える。

## 25. まとめ

curacuri.ai は、競合が存在する領域に入るプロダクトである。

しかし、その競合の多くは以下のいずれかに寄っている。

- AI FAQ Bot
- AIサポート
- チケット管理
- モデレーション
- 汎用Discord Bot
- キャラクター会話Bot
- 人力運営代行

curacuri.ai が狙うのは、その中間にある未成熟な領域である。

> 日本語公式Discord向けの、AI書記・受付・運営補佐キャラクター。

v2では、特に以下を重視する。

- 公式Discord運営を「小規模・長期運営の認知負荷問題」として捉える
- MVPの中心体験を「1週間分のDiscordを5分で把握する」に置く
- 週次運営レポートを最重要機能にする
- OSSだけでなくHosted FreeをMVPから用意する
- 分類精度を定量検証する
- 監視感ゼロ設計を初期機能に含める
- 最初のターゲットをAIプロダクト公式Discordに絞る
- 早期収益化は運営代行会社向けB2Bも重視する
- APIコスト・セットアップ負荷・プライバシー対応を最初から設計に含める
- 「第二の脳」は初期訴求ではなく、長期ビジョンとして扱う

売りは業務価値。  
中心体験は認知負荷低減。  
導入障壁を下げるのはHosted Free。  
継続利用の理由はキャラクター性。  
初期収益化の候補はAgency/Team。  
長期的な独自性は共同体記憶。

curacuri.ai は、公式Discordを「流れて消えるチャット」から、「ユーザーの声が蓄積され、運営判断に接続される共同体の記憶」へ変える。

そして、小規模運営者が長く続けられるための、AI時代のコミュニティ運営インフラを目指す。
